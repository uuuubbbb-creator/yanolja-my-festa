# Project Overview

**한 줄 소개**: 전국 1,873개 지역축제 중 '나에게 맞는 축제'를 BERTopic 기반으로 진단·추천하고, 숙박·교통·레저 예약까지 한 번에 끝내는 야놀자 신규 카테고리 제안.

| 항목 | 내용 |
|---|---|
| 기간 | 2026년 트래블 이노베이션 아이디어 공모전 |
| 팀 구성 | 4인 (문은비, 박현지, 송윤지, 황민우) |
| 담당 역할 | 팀 리더 · 전 과정 주도 (문제정의 → 토픽모델링 → 추천 알고리즘 → 서비스 기획 → 발표) |
| 사용 기술 | Python, Kiwi(형태소분석), LDA, CTM, BERTopic(SBERT+UMAP+HDBSCAN), FastAPI, Selenium |
| 핵심 성과 | 지배 토픽 확률 0.616 (+0.165), 다중 토픽 비율 42.0%p 개선, 축제+숙소+교통+레저 통합 패키지 추천 시스템 구현 |

# 1. Problem — 왜 지금 지역축제인가

## 1-1. 시장은 역대 최대, 그런데도 외면받는 축제들

[📷 상황 분석 — 전국 지역축제 연간 900개+ / 관광소비지출·관광객수 그래프]

한국문화관광연구원 자료에 따르면 축제 기간 중 관광소비지출은 비축제 기간 대비 **+6.5%**, 관광객 수는 **+19.5%** 증가합니다. 지역축제는 이미 지역경제를 움직이는 핵심 콘텐츠입니다.

문제는 그 다음입니다.

[📷 상황 분석 — 축제별 방문객 쏠림 분포, 상위 10% 기준선]

| 지표 | 수치 |
|---|---|
| 2026년 전국 지역축제 수 | 1,266개 (역대 최고) |
| 총 방문객 | 1.6억 명 (4년 만에 약 5배 상승) |
| 상위 10% 축제가 가져가는 방문객 비중 | **64%** |
| 하위 70% 축제가 나눠 갖는 방문객 비중 | 단 **12%** |

> **시장 자체는 커지고 있지만, 대부분의 지역축제는 여전히 외면받고 있다.**

## 1-2. 소비를 이끌 MZ가 오지 않는다

[📷 상황 분석 — 방문객 연령 구성 파이차트]

2025년 문화관광축제 21개 기준, 핵심 방문층은 **60대 이상(28.6%)**이고 2030세대는 합쳐서 26.2%에 그칩니다. 그런데 MZ는 여행 빈도가 높고 소비에 적극적이며 SNS 확산력이 강해, 지역축제 활성화의 핵심 타깃일 수밖에 없습니다.

여기서 저희 팀이 처음 세운 가설은 **"MZ가 지역축제 자체에 관심이 없다"**는 것이었습니다. 하지만 광안리 방어축제처럼 인지도가 있는 축제는 MZ 비중이 **69%**까지 올라갑니다. 조건은 하나였습니다 — SNS에서 봤거나, 친구가 다녀왔거나, 이름을 들어본 곳. 즉 **"검증된 소수 축제만 선택하는 구조"**였습니다.

> 가설이 기각되고 새로운 질문이 생겼습니다: MZ가 안 가는 게 아니라, **못 찾는 것 아닐까?**

# 2. Research — 가설을 데이터로 검증하다

## 2-1. "다 비슷해 보인다"는 감(感)을 데이터로 확인

[📷 문제 정의 — "다 비슷해 보인다" vs "실제로는 서로 다른 경험 유형" 비교 테이블]

검색창에 '축제'를 치면 목록은 쏟아지지만 차별점이 보이지 않습니다. 세계축제연구소장과 대학생 인터뷰에서도 "비슷비슷한 축제가 너무 많다", "특색 없는 축제는 가고 싶지 않다"는 반응이 반복됐습니다.

이 지점에서 저희는 **직접 검증**하기로 했습니다. "정말 축제들이 다 비슷한가, 아니면 차이는 있는데 안 읽히는가?" — 이 질문에 답하려고 전국 1,922개 축제의 설명 텍스트를 BERTopic으로 분석했습니다.

결과는 명확했습니다. 실제로는 **7개의 뚜렷하게 다른 경험 유형**(청년·가족 체험형 18.1%, 자연·계절 경관형 18.1%, 역사·문화유산형 17.2% 등)으로 나뉘었습니다.

> **차이는 분명히 존재한다. 다만 사용자 언어로 번역되지 않아 읽히지 않을 뿐이다.**

이 발견이 이후 모든 기술적 의사결정(토픽모델링을 왜 하는가)과 서비스 설계(왜 진단 퀴즈가 핵심 기능인가)의 근거가 됩니다.

## 2-2. 야놀자를 선택한 이유 — 이미 있는 수요에 올라타기

[📷 야놀자 '지역축제' 탭 신설 — MAU/비수도권 예약 비중/생산유발 효과]

새 플랫폼을 만드는 대신 야놀자를 택한 이유도 데이터로 뒷받침했습니다.

- MAU 324만 명, 비수도권 숙소 예약 비중 2019년 55% → 2024년 60%로 이미 지역 체류형 여행 수요 보유
- KTX+숙소 패키지 출시 후 예약 건수 전년 대비 **+38%** 급증 → 여행 수단 제공이 실제 방문 수요를 만든다는 근거
- 문화관광축제 외지인 방문객 소비 비중 **55.9%** → 축제는 실제 방문·체류 가능성이 높은 목적지

> 단순 정보 제공을 넘어, **실제 방문과 체류를 유도하는 지역축제 플랫폼**이 필요하다는 결론에 도달했습니다.

# 4. Development — 토픽모델링, 세 번 갈아엎은 이유

축제를 '유형'으로 묶는 작업은 한 번에 성공하지 않았습니다. 왜 각 모델을 버렸는지가 이 프로젝트의 기술적 핵심입니다.

## 4-1. 시도 1 — LDA: 빠르지만 문맥을 모른다

[📷 축제 데이터 토픽모델링 (1) LDA — Coherence/Perplexity 그래프, K=10 토픽 결과]

- **입력**: 축제 소개문 → Kiwi 형태소 분석 → BoW 행렬
- **선정 과정**: Coherence 최대·Perplexity 최소 지점을 찾아 K=10으로 확정
- **결과**: 10개 유형이 도출됐지만 지배토픽 평균 확률이 0.563에 그쳐, 상당수 축제가 복수 토픽에 걸쳐 있었다

**한계**: LDA는 BoW(단어 빈도) 기반이라 어순과 문맥을 반영하지 못합니다. "청년들이 즐기는 예술 공연"과 "예술가들이 즐기는 청년 공연"을 같은 벡터로 취급하는 구조적 한계가 있었습니다.

> → 문장 수준 의미를 포착하는 **CTM**으로 전환

## 4-2. 시도 2 — CTM: 문맥은 잡았지만 라벨링이 주관적

[📷 축제 데이터 토픽모델링 (2) CTM — 모델 구조, K=9 토픽 결과, Coherence 0.439]

CTM(Combined Topic Models)은 SBERT 문장 임베딩(의미)과 BoW(빈도) 표현을 함께 인코더에 넣어 잠재 벡터를 학습하는 딥러닝 모델입니다. `paraphrase-multilingual-MiniLM-L12-v2` 임베딩을 사용해 K=9에서 Coherence 최대(0.4407)를 확인했습니다.

**한계**: 정량 성능은 확보했지만, 토픽 수를 몇 개로 정할지·각 토픽에 어떤 이름을 붙일지가 여전히 분석가의 주관적 판단에 크게 의존했습니다.

> → 사전학습 언어모델로 토픽을 직접 추출하는 **BERTopic**으로 최종 전환

## 4-3. 시도 3 — BERTopic: 성공, 그러나 또 다른 함정

BERTopic은 "문서 군집화"와 "토픽을 설명하는 키워드 추출"을 완전히 분리합니다. 학습 시에는 SBERT로 문맥을 온전히 활용하고, 키워드 추출 단계에서는 형태소 분석기를 거친 문서로 c-TF-IDF를 계산합니다.

### 전처리 — 도메인 지식을 코드로 박아넣기

축제 설명문은 지역명·브랜드명·행정 상투어가 많아 그대로 넣으면 노이즈가 큽니다. 아래는 실제 사용한 전처리 함수의 핵심부입니다.

```python
def clean_text_for_bertopic(text: str) -> str:
    text = str(text)
    # [1] 브랜드명 → 의미 범주 치환 (긴 것부터) : 이월드 → '놀이공원'
    text = apply_brand_replace(text, brand_replace_dict)
    # [2] 지역명 제거 (긴 것부터) : 시/도/시군구 130여 개 목록
    for phrase in region_stopwords:
        text = text.replace(phrase, ' ')
    # [3~5] URL/특수문자 제거, 공백 정리
    text = re.sub(r'https?://\S+|www\.\S+', '', text)
    text = re.sub(r'[^가-힣\s]', ' ', text)
    text = re.sub(r'\s+', ' ', text).strip()

    # [6~7] Kiwi 형태소 분석 → 명사/동사/형용사(2글자 이상)만 추출
    tokens = kiwi.tokenize(text)
    meaningful_words = []
    for t in tokens:
        word, pos = t.form, t.tag
        if pos in ('NNG', 'NNP', 'VV', 'VA') and len(word) >= 2:
            if word not in manual_stopwords:
                meaningful_words.append(word)
    return ' '.join(meaningful_words)
```

이 함수가 중요한 이유는 단순 전처리가 아니라 **도메인 판단이 코드로 내려앉은 지점**이기 때문입니다.

- `brand_replace_dict`: "이월드→놀이공원", "휴애리→정원"처럼 개별 브랜드명을 상위 의미 범주로 치환해, 토픽이 특정 브랜드가 아니라 '경험의 종류'로 묶이도록 유도
- `region_stopwords`: 시/도/시군구 이름을 제거하지 않으면 토픽이 지역별로 갈라져버려, 정작 원하는 '경험 유형' 군집이 나오지 않음
- 8야 행정 용어("야로", "야사" 등 '~야행' 파생어) 제거 vs "야식", "야경" 등 실제 의미어 유지 — 단위 테스트 5종으로 검증 후 진행

전처리 품질은 매 단계 체크리스트로 검증했습니다: 어휘 다양성(TTR) 0.15 이상, 문서 생존율 95% 이상, 품사 태그 잔존 없음 등 20개 항목을 모두 통과한 뒤에야 임베딩 단계로 넘어갔습니다. 이렇게 "검증 없이 다음 단계로 넘어가지 않는다"는 원칙을 전 파이프라인에 일관되게 적용했습니다.

### 4-4. 유형 테스트 모델의 첫 실패 — 확률 체계 불일치

[📷 BERTopic-유형 테스트 — 문제 발생 및 원인 분석: 밀도기반 확률 vs 거리기반 재배정 괴리]

```python
umap_model = UMAP(n_neighbors=5, n_components=5, min_dist=0.0,
                   metric='cosine', random_state=42)
topic_model = BERTopic(embedding_model=sentence_model, umap_model=umap_model,
                        min_topic_size=30, nr_topics=8)
topics, _ = topic_model.fit_transform(docs, embeddings=embeddings)

new_topics = topic_model.reduce_outliers(
    docs, topics, strategy="embeddings", embeddings=embeddings
)
```

목표 토픽 수를 8개로 강제 지정하는 과정에서 소규모 토픽은 재배정, 대규모 토픽은 K-means로 분할했는데, **최종 배정된 지배 토픽과 축제별 토픽 부합률이 서로 맞지 않는 오류**가 발생했습니다.

원인을 추적한 결과: 초기 확률(`probs`)은 HDBSCAN의 **밀도 기반** 소프트 클러스터링 확률인 반면, 재배정/분할 단계는 K-means 중심점과의 **거리 기반** 코사인 유사도로 계산되고 있었습니다. 서로 다른 두 확률 체계가 섞여 있었던 것입니다.

**해결**: `calculate_probabilities=False`로 밀도 기반 확률을 아예 제거하고, 모델 생성/재배정/분할이 끝난 뒤 **최종 토픽 중심점과 전체 임베딩 벡터 간 코사인 유사도**로 일괄 재계산했습니다. 매 단계를 동일한 유사도 체계로 통일한 것입니다.

**결과**: 8개 유형(겨울/크리스마스형, 자연/계절형 등) — Coherence 0.515, Topic Diversity 0.875, Silhouette 0.0735. 이 모델은 "당신은 OOO 유형입니다"라는 직관적 결과를 주는 **유형 진단(퀴즈)** 용도로는 충분했습니다.

### 4-5. 유형 테스트 모델의 두 번째 실패 — 추천에는 못 쓴다

[📷 BERTopic-축제 추천 — 한계1: 다주제 성격 증발, 한계2: 아웃라이어 폭발]

유형 테스트 모델을 그대로 **추천**에 쓰려 했을 때 새로운 문제가 드러났습니다.

- **Hard Assignment의 함정**: 모든 축제를 지배 토픽 1개(100%)에 강제 배정 → 지배 토픽 부합도 평균 0.4217, 그중 65.9%가 사실 여러 토픽에 걸쳐 있는 '다중 토픽 축제'였음. "봄꽃 축제이면서 동시에 지역 특산물 축제"인 케이스가 한 카테고리로 뭉개짐
- **아웃라이어 폭발**: 파라미터를 정상화(min_dist를 0이 아니게, 등)하면 **40%의 축제**가 어느 토픽에도 속하지 못하고 `-1`(아웃라이어) 판정됨

> 8개의 거시적 카테고리는 "당신은 무슨 유형입니까"에는 적합했지만, "이 축제와 저 축제 중 뭐가 더 잘 맞습니까"라는 정교한 추천 질문에는 답할 수 없었습니다.

### 4-6. 최종 해법 — Soft Assignment + Two-track 구조

[📷 BERTopic-축제 추천 — Hard→Soft Assignment 전환, 매크로 7개 테이블]

핵심 아이디어 전환: **하나의 축제를 하나의 토픽에 강제로 넣지 말고, 전체 토픽에 대한 확률 벡터를 통째로 저장하자.**

```python
# HDBSCAN 원본 확률(probs) → -1(아웃라이어) 컬럼 제거 후 행 재정규화
probs_arr = np.array(probs)
valid_mask = [i for i, t in enumerate(topic_ids_all) if t != -1]
probs_no_neg = probs_arr[:, valid_mask]

row_sums = probs_no_neg.sum(axis=1)
row_sums[row_sums == 0] = 1.0
soft_matrix = probs_no_neg / row_sums[:, None]   # 문서 수 × 토픽 수, 행합=1
```

여기에 51개의 세밀한 마이크로 토픽을 먼저 추출한 뒤, 팀 검토를 거쳐 7개 매크로 카테고리로 병합하는 **Two-track 구조**를 도입했습니다.

```python
MACRO_MAP = {
    'M1_자연계절':     [0, 10, 16, 19, 26, 30, 34, 38, 41],
    'M2_역사문화':     [2, 11, 12, 14, 22, 32, 35, 37, 42, 46, 50],
    'M3_먹거리특산물': [1, 4, 36, 43, 44, 45, 47],
    'M4_예술공연창작': [6, 18, 23, 27, 29, 33, 39, 48],
    'M5_가족체험':     [5, 13, 15, 20, 21, 24, 25, 40],
    'M6_청년마켓':     [8, 9, 28, 31],
    'M7_과학레저':     [3, 7, 17, 49],
}
# 마이크로 토픽 확률을 매크로 단위로 합산 후 재정규화
for m_idx, (macro_name, micro_list) in enumerate(MACRO_MAP.items()):
    for micro_tid in micro_list:
        macro_soft_matrix[:, m_idx] += soft_matrix[:, tid_to_col[micro_tid]]
macro_soft_matrix /= macro_soft_matrix.sum(axis=1, keepdims=True)
```

**병합 기준이 애매한 경계 사례**(예: Topic20 "궁중/웰니스/온천/한방" → M2 역사문화 vs M5 가족체험)는 감이 아니라 AB 테스트로 결정했습니다. Silhouette, 지배확률 평균, 평균 엔트로피, 다중 토픽 비율 4개 지표를 두 후보군(A=M5 배정, B=M7 배정)에 대해 비교해, 근소하게 우세한 쪽을 채택했습니다.

**최종 성과**:

| 지표 | 개선 전 (Hard, 51개) | 개선 후 (Soft, 7개 매크로) |
|---|---|---|
| 지배 토픽 확률 평균 | 0.4502 | **0.6156 (+0.165)** |
| 다중 토픽 문서 비율 | 63.4% | **42.0% (-21.4%p)** |
| 평균 엔트로피 | 2.3765 | **1.1015 (-1.275)** |

[📷 매크로 별 평균 확률 히트맵 / 매크로 간 유사도 히트맵 — 대각선 우세로 자기일치도 확인]

> 단일 유형 분류의 한계를 벗어나, 축제의 복합적인 매력을 확률 분포 그대로 계산에 반영하는 모델을 완성했습니다. 이 `soft_matrix`가 이후 취향 진단 퀴즈와 맞춤 추천 두 기능 모두의 공통 기반이 됩니다.

```python
def split_budget_and_special(packages, budget_min, budget_max):
    ranked = rerank_packages_for_budget(packages, budget_max)
    within_pool  = [p for p in ranked if budget_min <= p["total_sale_price"] <= budget_max]
    special_pool = [p for p in ranked
                     if budget_max < p["total_sale_price"] <= int(budget_max * SPECIAL_BUDGET_MULTIPLIER)]
    # within → special 순서로 다양성 카운터를 이어받아 통합 cap 적용
    # (동일 숙소/레저가 조합에 반복 등장하지 않도록 최대 2회로 제한)
    within  = _apply_diversity_filter(within_pool,  BUDGET_TOP_N,  lodging_counter, leisure_counter)
    special = _apply_diversity_filter(special_pool, SPECIAL_TOP_N, lodging_counter, leisure_counter)
    return within, special
```

**왜 다양성 필터가 필요했는가**: 초기 버전은 점수 순으로만 잘랐더니, TOP7 패키지 대부분이 "같은 숙소 + 다른 레저 조합"으로 도배되는 문제가 있었습니다. 사용자 입장에서는 사실상 선택지가 1~2개뿐인 셈이었습니다. `LODGING_DIVERSITY_CAP`, `LEISURE_DIVERSITY_CAP`으로 동일 상품의 반복 등장을 제한해, 진짜 '7가지 다른 선택지'가 나오도록 만들었습니다.

1. **데이터 한계**: 1,873개의 정적 데이터, 텍스트 자체가 빈약한 축제(1~2문장)는 BERTopic 토픽 벡터 신뢰도가 낮습니다. 소규모 지역축제일수록 이 문제가 두드러집니다.
2. **추천-예약 연동 한계**: 현재는 정적 가격·재고 데이터를 사용하므로, 실 서비스에서는 실시간 재고 연동과 외부 결제 이탈 방지가 추가로 필요합니다.
3. **비즈니스 모델 확장**: 현재의 B2B 단일 모델에서 유저·지자체·야놀자 3축 수익 구조로 발전시킬 여지가 있습니다.

> 모델을 세 번 갈아엎은 이유도, 매크로 토픽 병합에 AB테스트를 쓴 이유도 결국 하나입니다 — **"그럴듯해 보이는 결과"와 "실제로 작동하는 결과"를 구분하려는 태도**입니다. 이 프로젝트에서 가장 크게 배운 것도 그 지점이었습니다.
